APPENDIX A — FIRST LIVE SIGNUP TEST: WHAT BROKE AND WHY
===============================================================================
After deploying the marketing site but before finishing Step 1 of this
guide, a real signup attempt surfaced three separate issues worth
distinguishing clearly:

1. WORKER NOT RESOLVING (net::ERR_NAME_NOT_RESOLVED on /api/profile)
   This is a DNS-level failure, not a 404 — it means the worker named
   my-sniper-worker doesn't currently exist under your Cloudflare account
   at all (or its workers.dev route is disabled). This is why the dashboard
   fails to load right after signup even though the Supabase account itself
   gets created fine (plain email signup talks to Supabase directly, not
   your worker — it's only the very next step, loading the profile, that
   needs the worker).
   FIX: Complete Step 1 of this guide (deploy my-sniper-worker for real),
   then run:
     curl -I https://my-sniper-worker.daniellancce1.workers.dev/debug/env
   If curl can't resolve the host at all, go to Cloudflare Dashboard >
   Workers & Pages and confirm "my-sniper-worker" is actually listed. If
   it's listed, open it > Settings > Domains & Routes and confirm the
   workers.dev route is enabled.

2. SOCIAL LOGIN BUTTONS (Google/Facebook/LinkedIn/Instagram all failing
   with "Unsupported provider: provider is not enabled")
   This is NOT a code bug — handleOAuth() in index.html is correct. None of
   these four providers have been turned on yet in Supabase.
   FIX, per provider, in Supabase Dashboard > Authentication > Providers:
     a. Create OAuth credentials in that platform's own developer console
        (Google Cloud Console / Facebook for Developers / LinkedIn Developer
        Portal — Instagram piggybacks on the Facebook app).
     b. Use the callback URL Supabase shows on that same Providers page as
        the redirect URI when registering the app on the provider's side.
     c. Paste the resulting Client ID + Client Secret into Supabase and
        toggle the provider on.
   ALSO REQUIRED (separate from the above, applies to all providers at
   once): Authentication > URL Configuration > both Site URL and Redirect
   URLs must include exactly:
     https://626gl1ch.github.io/JOB-FINDER-SAAS-WEB-APP/
   (trailing slash included) — a mismatch here causes a different failure
   even after the provider itself is enabled correctly.
   Recommendation: get Google working first end-to-end, then repeat for the
   others. If you don't need Instagram sign-in soon, it's fine to remove
   that button rather than configure all four before launch.

3. MISSING HERO IMAGE (hero-bg.jpg 404)
   index.html references assets/hero-bg.jpg, but no assets/ folder exists
   in the repo. Either add a real image at that exact path, or remove that
   <img> tag if you don't have one ready — purely cosmetic, not blocking.

EXTENSION NOISE YOU CAN IGNORE
   Console lines from "contentscript.js" mentioning ObjectMultiplex,
   MaxListenersExceededWarning, or "Could not establish connection" are
   from a browser extension (most likely a crypto wallet like MetaMask)
   injecting itself into every page you visit — unrelated to SnipeJob.
   Confirm by testing in an Incognito window with extensions disabled.

APPENDIX B — TROUBLESHOOTING
===============================================================================
"debug/env shows false for one of the keys"
  → You ran the wrangler secret put command in the wrong folder. cd into
    my-sniper-worker first, then re-run the command for that specific key.

"Routes still say Not Found after deploying"
  → You're deploying the wrong file. Confirm my-sniper-worker/wrangler.jsonc
    has "main": "src/index.js" and that you ran `npx wrangler deploy` from
    INSIDE the my-sniper-worker folder, not the project root.

"Database error saving new user" during signup
  → Re-run the ORIGINAL schema.sql (not just the additions from Step 3) in
    Supabase SQL Editor — this re-applies the handle_new_user() trigger fix.

"AI features return a 502 / temporarily unavailable"
  → Your GEMINI_API_KEY secret is missing, wrong, or you've hit a Gemini
    quota limit. Check Cloudflare Dashboard > Workers > my-sniper-worker >
    Logs (Real-time Logs) while reproducing the error to see the exact
    Gemini error message.

"Trends panel is empty even after Step 3"
  → The first request for a sector triggers Gemini generation, which takes
    a few seconds — wait and refresh. If still empty after that, check the
    real-time Worker logs for a Gemini error.

"Earnings still show empty after Step 3"
  → Double-check the SQL in Step 3 actually ran without error (Supabase SQL
    Editor shows a green "Success" message). Re-run the policy check query
    from Step 3's verification.
